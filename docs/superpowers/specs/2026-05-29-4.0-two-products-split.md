# FormFlow 4.0 — IntelliSOURCE / Builder structural split

> Author: Claude (synthesizing CAT's audit + the day's pain)
> Date: 2026-05-29
> Status: **Spec, not implementation** — held for a forcing function (see §10).
> Supersedes Phase 6 of `2026-05-28-post-audit-cleanup.md`.

## 1. Goal

Make the boundary between **IntelliSOURCE Wizard** (the hardcoded 5-step
utility enrollment flow) and **Generic Form Builder** (the
FormRenderer-driven custom form path) explicit, architectural, and
honest — so a future change to one cannot accidentally break the other,
and so the plugin can be sold to non-utility clients without inheriting
IntelliSOURCE vocabulary.

**Non-goal:** new features. 4.0 is a structural refactor, not a feature
release.

## 2. Why now is *not* now

The structural split is **technical debt insurance, not a feature**. Do
it when one of these forcing functions lands:

- A second utility client whose wizard doesn't fit the IntelliSOURCE
  shape (different validation, different scheduling, different state
  machine).
- A non-utility client buys FormFlow Pro and is confused by the
  "Smart Meter" / "device_type" / "account number" vocabulary.
- A new engineer onboarding has to learn both code paths simultaneously.
- The legacy `admin/views/instance-editor.php` (1,300 lines) becomes
  blocking for a feature we want to ship.

If none of those land within 90 days, this spec stays a spec.

## 3. Current state — the seam

### Two products, one row

`wp_isf_instances` carries every form regardless of subsystem. The
`form_type` ENUM today:

```sql
ENUM('enrollment','scheduler','external','custom')
```

- `enrollment` + `scheduler` → IntelliSOURCE wizard path
- `external` → tracked redirect button (a third path, both subsystems can use)
- `custom` → builder path

The submissions table (`wp_isf_submissions`) has IntelliSOURCE-flavored
columns (`account_number`, `device_type`) that are empty for every
builder submission.

### Code paths almost never overlap

| Pipeline | Files | Shared with the other? |
|---|---|---|
| IntelliSOURCE wizard | `public/class-public.php::render_form_shortcode`'s legacy fallthrough; `public/templates/enrollment/step-{1..5}-*.php`; `isf_submit_enrollment` AJAX; `isf_validate_account`; `isf_get_schedule_slots`; `isf_book_appointment` | Database, Encryption, Security, FeatureManager, FormHandler-base |
| Builder | `public/class-public.php::render_custom_form`; `includes/builder/class-form-renderer.php`; `includes/builder/class-form-builder.php`; `public/assets/js/builder-form.js`; `isf_submit_builder_form` AJAX | Database, Encryption, Security, FeatureManager, Destinations, FormHandler-base |

### What the audit found

CAT's bold call: *"This plugin is two products."*

Every today's regression (3.0.5 / 3.0.7 / 3.1.1 / 3.1.4 / 3.1.5) came
from the seam between them — the public renderer dispatching the wrong
branch, hooks firing with the wrong signature for the wrong subsystem,
the form-editor's task views applying IntelliSOURCE assumptions to
builder forms.

## 4. Target architecture

One plugin, three directories at the top of `includes/`:

```
includes/
  intellisource/                     ← IntelliSOURCE-specific
    class-wizard-renderer.php         (moved from public/class-public.php fallthrough)
    class-account-validator.php       (moved from public/traits/trait-ajax-handlers.php)
    class-scheduler-api.php           (moved from same)
    templates/
      step-1-program.php
      step-2-verify.php
      step-3-info.php
      step-4-schedule.php
      step-5-confirm.php
    isf-wizard-strings.php            (centralized IntelliSOURCE vocab for i18n)

  builder/                            ← Generic builder (mostly already there)
    class-form-renderer.php
    class-form-builder.php
    class-conditional-logic.php
    public/builder-form.js
    (existing file layout, no major changes)

  shared/                             ← BOTH use these
    Database (wp_isf_instances + wp_isf_submissions + traits)
    Encryption
    Security
    FeatureManager
    Hooks (constants used by both subsystems)
    Destinations (subsystem-agnostic delivery pipeline)
    FormHandler-base (validation primitives)
```

### Dispatch — one entry point, two routes

```php
// public/class-public.php
public function render_form_shortcode($atts) {
    $instance = $this->db->get_instance_by_slug($atts['instance']);
    if (!$instance) return $this->not_found();

    switch ($this->classify($instance['form_type'])) {
        case 'external':       return $this->render_external_form($instance, $atts);
        case 'intellisource':  return (new IntelliSource\WizardRenderer($instance))->render($atts);
        case 'builder':        return (new Builder\FormRenderer($instance))->render($atts);
    }
}

private function classify(string $form_type): string {
    return match ($form_type) {
        'enrollment', 'scheduler',
        'intellisource_wizard', 'intellisource_scheduler' => 'intellisource',
        'custom', 'builder'                               => 'builder',
        'external'                                        => 'external',
        default                                           => 'builder',
    };
}
```

Old `form_type` values keep working via the classify normalizer. New
values are canonical; sites can migrate at their own pace.

## 5. `form_type` rename

| Today's value | 4.0 canonical value | Backwards-compat? |
|---|---|---|
| `enrollment` | `intellisource_wizard` | ✅ old value classified as intellisource forever |
| `scheduler` | `intellisource_scheduler` | ✅ same |
| `custom` | `builder` | ✅ old value classified as builder forever |
| `external` | `external` | unchanged |

### Migration SQL

```sql
-- New ENUM (additive; old values preserved)
ALTER TABLE wp_isf_instances MODIFY COLUMN form_type
  ENUM('enrollment','scheduler','external','custom',
       'intellisource_wizard','intellisource_scheduler','builder')
  DEFAULT 'builder';
```

The default flips from `'enrollment'` to `'builder'` because the
common case for new clients is the generic builder. IntelliSOURCE
becomes opt-in.

**Existing rows are not rewritten.** A 5-year-old Itron instance keeps
`form_type='enrollment'` forever; the classifier handles it. We optionally
provide a one-shot `wp formflow migrate-form-types` WP-CLI command that
rewrites rows to canonical names; it's never required.

## 6. PR sequence

Each PR is shippable on its own and protected by Phase 5 tests.

### PR 1 — Reorganize files (no logic changes)

- `git mv` IntelliSOURCE files into `includes/intellisource/`
- `git mv` builder files into `includes/builder/` (mostly already there)
- Update namespace declarations + autoloader rules
- Phase 5 `RequireOnceTargetsExistTest` catches any stranded require_once
- Phase 5 `ShortcodeDispatchTest` catches any dispatch regression
- **Ships independently** — no behavior change visible to users

### PR 2 — Add ENUM + classifier

- Add the new ENUM values via `run_migrations()`
- Add `Frontend::classify()` normalizer
- Update `ShortcodeDispatchTest` to assert both old and new values
  route to the right pipeline
- Update `FormTypeEnumRoundTripTest` to include new values in the
  canonical set
- **Ships independently** — both old and new values work

### PR 3 — Admin UI: subsystem-aware "create form"

- Replace the `form_type` dropdown in "create new form" with two big
  cards: **"IntelliSOURCE Enrollment Wizard"** + **"Custom Form Builder"**
- Add screenshots and a one-line description on each
- New forms are created with canonical names (`intellisource_wizard`,
  `builder`)
- Form Editor Beta task list reads from a per-subsystem registry —
  builder forms hide the "Connector" and "Scheduling" tasks
- **Risk:** new test required — `SubsystemTaskRegistryTest` asserts
  each subsystem exposes only the tasks it implements

### PR 4 — Deprecation notices

- Old `form_type` values still work, but `_doing_it_wrong()` notice
  fires on admin pages when one is read
- Notice points at the WP-CLI migration command
- Logged-out front-end is silent (no notice in production page renders)
- **Ships independently**

### PR 5 — Retire the legacy editor

- Delete `admin/views/instance-editor.php` (1,300 lines)
- Delete legacy menu registration + redirect-old-editor logic in
  `class-plugin.php`
- Form Editor Beta is no longer "Beta" — rename, promote
- ISF_NEW_EDITOR feature flag deleted
- **Risk:** must be last. Any old WPCode/cron/code that links to the
  legacy editor breaks. Pre-flight: grep for `isf-instance-editor` in
  all known consumer sites.

### Cumulative LOC

Estimated net change: **roughly neutral**. ~1,300 LOC deleted (legacy
editor) + ~500 LOC moved (no net change) + ~200 LOC added (classifier,
subsystem registry, screenshots). Mostly reorganization.

## 7. Phase 5 tests that protect this

| Test | What it catches |
|---|---|
| `FormTypeEnumRoundTripTest` | New ENUM values rejected by MySQL because someone forgot the migration |
| `ShortcodeDispatchTest` | Form_type normalization breaks; old `'enrollment'` no longer reaches the wizard |
| `RequireOnceTargetsExistTest` | `git mv` left dead require_once behind |
| `PublicRestRoutesDocumentedTest` | New subsystem-specific routes ship without auth-model documentation |
| **New** `SubsystemTaskRegistryTest` | A subsystem registers a task it doesn't actually render |
| **New** `LegacyAliasNormalizerTest` | The classifier doesn't normalize a legacy alias correctly |

## 8. Risk register

| Risk | Likelihood | Blast radius | Mitigation |
|---|---|---|---|
| Dominion's `form_type='custom'` form breaks during the rename | Low | Catastrophic | `ShortcodeDispatchTest` + classifier accepts old values forever |
| Itron-deployed enrollment wizards break | Low | High | Same — `'enrollment'` keeps working |
| License Server confusion (1 license today, 2 tiers tomorrow) | Medium | Low | Both tiers issue same license type initially; tier is a feature flag readable by Plugin |
| Legacy editor users (sites pinned to ISF_NEW_EDITOR=false) break in PR 5 | Medium | Medium | Pre-flight grep on all consumer sites; pin those sites to 3.x until they migrate |
| ENUM migration silently truncates form_type on a multisite/HyperDB install (the 3.0.6 ghost) | Low | High | Phase 5 enum test + use `$wpdb->dbname` (already fixed in 3.2.1) |
| Plugin update process can't handle multi-PR rename | Low | Low | Each PR is independently shippable; no PR depends on the next one being deployed |

## 9. Open questions for Nat

1. **License Server tier strategy** — Pro Utility (both subsystems) vs.
   Pro Builder (builder only) sellable as separate SKUs? Or one
   undifferentiated Pro?
2. **`scheduler` form_type** — today this exists but is IntelliSOURCE-
   specific (uses the scheduler API). Does the *generic* builder ever
   need a scheduler form_type, or does scheduling live entirely under
   `intellisource_scheduler`?
3. **3.x maintenance window** — after 4.0 ships, how long do we
   backport security fixes to 3.x for sites that can't upgrade?
4. **`marketplace/` Templates UI** — does each subsystem have its own
   template gallery, or do all templates live in one mixed namespace?
5. **Legacy `instance-editor.php` deletion timing** — PR 5 or a later
   4.1 release after the new editor has soaked for a month?

## 10. Forcing functions

This spec sits dormant until one of these is true:

- [ ] A second client onboards who is not Itron / utility (forcing function: vocabulary)
- [ ] An engineer other than Nat needs to onboard to the codebase (forcing function: cognitive load)
- [ ] A second wizard-shape product is requested (forcing function: parallel state machines)
- [ ] A feature we want to ship is blocked by the legacy editor (forcing function: tactical)

When one fires, this spec becomes the writing-plans skill input. Until
then it stays here as the **conscious deferral** record so a future
session doesn't re-discover the architecture problem and re-solve it.

## 11. Out of scope for 4.0

- New field types in the builder
- Multi-step builder forms (today builder is single-step; multi-step is
  a 4.1+ feature)
- A complete UI overhaul of the legacy enrollment wizard (still uses
  step-1-program through step-5-confirm templates as-is, just lives
  in `intellisource/templates/` instead of `public/templates/`)
- Detangling the destinations subsystem from `wp_isf_submissions` schema
- Anything that requires a wp_isf_instances or wp_isf_submissions
  schema change beyond the ENUM additive
